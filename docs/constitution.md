# 项目宪法

这份文档定义本项目后续整理、扩展和维护时必须遵守的规则。它只描述项目自身，不混入项目之外的个人准备内容。

## 1. 项目定位

本项目是一个基于 LangGraph 的多智能体客服 MVP 项目，当前聚焦多智能体客服核心链路。

项目当前目标是完成核心链路并保持后续扩展边界清晰：

- supervisor 分类和条件路由。
- 多个专业 Agent 分工处理任务。
- MCP 外部工具调用。
- 对联样本的检索增强生成。
- Gradio 本地页面展示执行结果。

当前 MVP 主体功能已经完成。后续优化围绕工程化和稳定性升级，不继续无边界堆功能。

## 2. 版本边界

README 只记录 V1、V2 两个已完成版本。

- V1：多智能体路由基础版。
- V2：工具调用与检索增强版。

V3 只能作为后续优化想法出现，不能写成已完成版本。

推荐的 V3 方向：

- 固定依赖版本。
- 优化配置读取。
- 增强 supervisor 分类稳定性。
- 增加 Mock 测试。
- 改善外部工具异常提示。
- 补充架构图文档。

## 3. Agent 边界

每个 Agent 只能负责自己的核心职责：

- `supervisor`：只负责问题分类、路由判断和流程收束。
- `travel`：只负责旅游路线规划和地图工具调用。
- `couplet`：只负责对联输入处理、参考样本检索和下联生成。
- `joke`：只负责笑话生成。
- `other`：只负责无法处理问题的兜底回复。

新增能力时，必须新增明确节点、路由规则和必要测试，不能把所有逻辑堆进 supervisor 或单个大文件。

## 4. 文件职责

项目文件按职责拆分：

- `app.py`：Gradio 本地页面入口。
- `agents/`：专业 Agent 节点。
- `graph/`：状态定义和图构建逻辑。
- `services/`：模型、工具、检索和 trace 展示等共享服务。
- `scripts/`：辅助运行脚本，不放核心业务逻辑。
- `config/`：Key 读取、模型配置和路径处理。
- `resource/`：本地数据资源。
- `tests/`：单元测试。
- `docs/`：项目文档。

根目录只保留真正入口文件和项目说明文件。临时测试脚本、辅助脚本、可复用逻辑都不能散落在根目录。

每个 Python 文件顶部必须有一句简短备注，说明该文件是做什么的。备注要短，不写长篇背景。

## 5. 工具与检索

外部依赖不能成为项目唯一运行路径：

- MCP 工具调用失败时，要有明确异常信息或兜底策略。
- Redis 不可用时，对联检索必须能降级到本地内存相似度计算。
- 本地资源路径必须通过 `config/paths.py` 解析，不能依赖当前运行目录。

## 6. 配置安全

真实密钥不能提交到仓库。

配置规则：

- `config/Keys.example.json` 只放示例 key。
- `config/Keys.json` 用于本地真实 key，必须保持私密。
- 模型名称集中放在 `config/model_config.py`。
- 新增 key 或模型配置时，要同步更新 README。

## 7. 测试原则

每次整理或修改核心流程后，至少运行：

```bash
python3 -m unittest discover tests
```

新增核心能力时，优先补不依赖真实大模型的测试，例如：

- 路径解析测试。
- trace 展示测试。
- 路由函数测试。
- 图构建测试。
- Redis 降级逻辑测试。

真实 LLM、MCP 或外部 API 测试可以作为手动集成验证，不应阻塞基础单元测试。

## 8. 文档原则

README 只写项目事实、架构、启动方式和真实能力。

文档中必须区分：

- 已完成能力。
- 后续优化想法。
- 暂未实现能力。

不能为了让项目看起来更大，把规划能力写成当前能力。
